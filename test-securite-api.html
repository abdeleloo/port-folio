Test de Sécurité API

Période : Mars 2026 – Avril 2026  |  Contexte : Alternance – IP Twins

Dans le cadre de mon alternance chez IP Twins, j'ai réalisé une campagne de tests de sécurité sur les APIs de l'application métier. L'objectif était d'identifier les vulnérabilités potentielles et de proposer des corrections pour renforcer la sécurité des échanges de données.

1

Contexte et objectifs

IP Twins expose plusieurs API REST utilisées par l'application interne pour gérer les domaines, les contacts et la facturation. Une montée en compétences sur la sécurité applicative était nécessaire avant l'ajout de nouvelles fonctionnalités.

Objectif 1

Identifier les failles de sécurité existantes

Objectif 2

Tester la robustesse des endpoints

Objectif 3

Rédiger un rapport de préconisations

2

Méthodologie et outils utilisés

Tests réalisés selon les recommandations OWASP Top 10 pour les API :

Outils utilisés

  • Postman – Tests manuels des endpoints
  • Burp Suite – Analyse du trafic HTTP
  • curl – Tests en ligne de commande
  • OWASP ZAP – Scan automatisé des vulnérabilités

Types de tests

  • Authentification et gestion des tokens JWT
  • Contrôle d'accès (BOLA / BFLA)
  • Injection SQL via paramètres API
  • Exposition excessive de données
  • Rate limiting et brute force

3

Résultats des tests

Exemples de vérifications réalisées (données anonymisées) :

Endpoint testé Type de test Résultat Criticité
/api/contacts Contrôle d'accès (sans token) Accès refusé (401) OK
/api/factures/{id} BOLA – accès à l'ID d'un autre utilisateur À corriger Haute
/api/login Brute force (sans rate limiting) Partiellement protégé Moyenne
/api/search Injection SQL (paramètre q) Requêtes préparées OK OK

Exemple de test – Vérification d'authentification

# Test 1 : accès sans token → doit retourner 401
curl -s -o /dev/null -w "%{http_code}" \
  https://api.iptwins.net/v1/contacts
# Résultat attendu : 401 Unauthorized

# Test 2 : accès avec token valide → doit retourner 200
curl -s -o /dev/null -w "%{http_code}" \
  -H "Authorization: Bearer <TOKEN>" \
  https://api.iptwins.net/v1/contacts
# Résultat attendu : 200 OK

# Test 3 : BOLA → accès à la ressource d'un autre utilisateur
curl -s -w "%{http_code}" \
  -H "Authorization: Bearer <MON_TOKEN>" \
  https://api.iptwins.net/v1/factures/9999
# Résultat observé : 200 OK ← vulnérabilité identifiée

4

Préconisations et corrections

  • BOLA : Ajouter une vérification côté serveur que l'ID demandé appartient bien à l'utilisateur authentifié avant de retourner la ressource.
  • Rate limiting : Implémenter un mécanisme de limitation des requêtes sur le endpoint /api/login (ex : 5 tentatives / 15 min par IP).
  • Tokens JWT : Vérifier l'expiration des tokens et mettre en place un mécanisme de révocation.
  • Logs de sécurité : Journaliser les tentatives d'accès non autorisées pour permettre une détection d'intrusion.
