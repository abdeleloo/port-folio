Test de Sécurité API

Période : Mars 2026 – Avril 2026  |  Contexte : Alternance – IP Twins

Dans le cadre de mon alternance chez IP Twins, j'ai réalisé une campagne de tests de sécurité sur les APIs de l'application métier. L'objectif était d'identifier les vulnérabilités potentielles et de proposer des corrections pour renforcer la sécurité des échanges de données.

1

Contexte et objectifs

IP Twins expose plusieurs API REST utilisées par l'application interne pour gérer les domaines, les contacts et la facturation. Une montée en compétences sur la sécurité applicative était nécessaire avant l'ajout de nouvelles fonctionnalités.

Objectif 1

Identifier les failles de sécurité existantes

Objectif 2

Tester la robustesse des endpoints

Objectif 3

Rédiger un rapport de préconisations

2

Méthodologie et outils utilisés

Tests réalisés selon les recommandations OWASP Top 10 pour les API :

Outils utilisés

  • Postman – Tests manuels des endpoints
  • Burp Suite – Analyse du trafic HTTP
  • curl – Tests en ligne de commande
  • OWASP ZAP – Scan automatisé des vulnérabilités

Types de tests

  • Authentification et gestion des tokens JWT
  • Contrôle d'accès (BOLA / BFLA)
  • Injection SQL via paramètres API
  • Exposition excessive de données
  • Rate limiting et brute force

Les tests ont été réalisés via Postman et curl en ligne de commande. Les extraits de commandes ci-dessus et le tableau de vulnérabilités constituent la documentation de l'audit.

3

Résultats des tests — Vulnérabilités identifiées

Audit réalisé sur 4 modules de l'application métier IP Twins (Domainarium II) :

Module Users

Problème Impact Criticité
Fuite de données : numéros de téléphone personnels et last_login_at exposés en clair Violation RGPD, données sensibles accessibles Haute
Profils complets exposés sans masquage des champs sensibles Répertoire complet du groupe accessible via un seul token compromis Haute

Module Tasks

Problème Impact Criticité
AuthCodes exposés en clair dans le JSON Risque de vol de domaines clients Critique
Paramètre include_user renvoie numéros de téléphone et e-mails du staff Exposition des données personnelles des employés Haute
Champ structure renvoie du code PHP brut dans la réponse JSON Exposition de la logique métier côté serveur Moyenne
Payload trop volumineux à cause des jointures Risque de saturation front-end et serveur Moyenne

Module LegalActions

Problème Impact Criticité
Erreur 500 (HTML Laravel) retournée à la place d'une 401 quand le token est manquant Exposition des détails de l'infrastructure interne Haute
Dossiers marqués status: ongoing alors qu'ils ont une end_at renseignée Données incohérentes, risque métier Moyenne
Paramètre type accepte n'importe quelle valeur libre sans validation Risque de pollution de la base de données Moyenne
Champ subject est une String JSON brute — force un JSON.parse() côté front Mauvaise pratique d'API, fragilité front-end Faible

Module Services

Problème Impact Criticité
Champs product_ref et extension : saisie libre sans whitelist, accepte caractères spéciaux et textes longs Risque d'injection, erreurs 500 imprévues Moyenne
/services/{id} — RAS Aucune vulnérabilité détectée OK
/legal-actions/{id} — RAS Aucune vulnérabilité détectée OK
4

Préconisations et corrections

  • Module Users : Appliquer un masquage des champs sensibles (phone, last_login_at) dans les réponses API — ne retourner que les champs nécessaires (principe du moindre privilège).
  • AuthCodes : Ne jamais exposer les AuthCodes dans les réponses JSON. Les chiffrer ou les transmettre uniquement via un canal sécurisé dédié.
  • Gestion des erreurs : Retourner systématiquement une réponse JSON normalisée (401, 403) au lieu de laisser Laravel exposer ses traces HTML en production.
  • Validation des entrées : Mettre en place une whitelist stricte pour les paramètres libres (type, product_ref, extension) et retourner une 422 sur entrée invalide.
  • Cohérence des statuts : Ajouter une règle métier côté serveur pour passer automatiquement un dossier de ongoing à closed lorsque end_at est renseigné.
  • Optimisation des payloads : Utiliser la pagination et limiter les jointures inutiles pour éviter la saturation du front-end et du serveur.
  • Rate limiting : Implémenter une limitation des requêtes sur /api/login (ex : 5 tentatives / 15 min par IP).
